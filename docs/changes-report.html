Release 0.9.1 - 2004-02-10

Type Changes By
Fixed #SJN-49, a problem in loading DataSources from particular .properties files. hen

Release 0.9 - 2004-02-07

Type Changes By
Tested, and fixed, the usage of http as a protocol for getting simple-jndi values, #SJN-27. Untested on servers that do not allow directory-listings. hen
Feature #SJN-30, to have a static memory behind the contexts and not a per-instance memory. It is turned on via the org.osjava.jndi.shared=true jndi.properties parameter. hen

Release 0.8.1 - 2004-01-18

Type Changes By
Added feature #SJN-48, in which the simple-jndi is configurable from system properties as well as jndi.properties and a Hashtable. The system property takes precedence. hen

Release 0.8 - 2004-01-03

Type Changes By
Fixed bug #SJN-45, in which the XML property loader did not correctly handle multiple tags with the same name forming a List object. hen
Fixed a bug in which 'ty pe' did not work in a DataSource file in which you had the DS name as the filename and not delimited in the file. hen
Deprecated usage of the 'org.osjava.jndi.datasource' property in favour of a simple type=javax.sql.DataSource. Especially as JIRA issue #SJN-42 exists with the datasource usage. Will be removed in 0.9. hen
Added an example for xml, ini and properties creation of Collections in simple-jndi. hen
Added an example for xml, ini and properties built DataSources in simple-jndi. hen
Connection pooling added via the use of Jakarta Commons DBCP. hen
While I've decided not to use log4j, I have implemented a classic System.out debug system with a -DSJ.DEBUG=true system property to turn it on. hen
Switched dependencies and inlined code so that no jars are needed by default, but to get the xml feature you need gj-xml, to get converting you need commons-convert and to get connection-pooling you need commons-dbcp, which needs commons-pool and commons-collections. hen

Release 0.7 - 2003-12-06

Type Changes By
XML bug #SJN-39 fixed. In 0.6, XML was barely functional and 0.7 fixes those bugs. hen
In addition to .properties and .xml files, simple-jndi now supports .ini files. The precedence in case of similar naming is .properties, .xml then .ini. hen
DataSources may now be configured through the '.type' conversion mechanism as well as the org.osjava.jndi.datasource mechanism. This is needed for XML which cannot use the org.osjava.jndi.datasource mechanism. Most likely this will be the only mechanism for DataSources in a future version. hen

Release 0.6 - 2003-08-12

Type Changes By
You may now see and modify the Simple-JNDI parameters through Simple-JNDI. hen
Because Windows cannot handle ':' in a filename, "java:" is quietly converted to "java". hen
Unit Tests written and plugged into Maven. hen

Release 0.5 - 2003-04-08

Type Changes By
Moved from the 'com.generationjava' namespace to the 'org.osjava' namespace, and an accompanying change in look and feel to the site. hen
Boolean and URL Converters added. hen
Values in the System properties [ie) -Dfoo.bar=baa ] now override simple-jndi. hen
Values may now be bound to the JNDI space, where they override the configured simple-jndi properties. When removed, the value goes back to the simple-jndi default. hen

Release 0.4 - 2003-02-27

Type Changes By
com.foo.Value can have a partner key of com.foo.Value.type which specifies the type of object to use for the key. While this system will be pluggable, currently it supports java.lang.Number only. hen
Any XML format may be used underneath instead of a .properties system. Properties files take precedence over XML files, so in the light of foo.xml and foo.properties, only foo.properties will be read. hen
The underlying properties parsing has been improved. It now handles ordered collections. These are implemented by specifying a key multiple times. One issue with this is that the object returned maybe a List or a String [or something else depending on type ] , leading to an ugly instanceof style of programming. The following eases this. hen
org.osjava.naming.Jndi is a wrapper which automatically grabs the default InitialContext and provides simple helper routines for getting data out. One such example is getList(String key) which improves the situation in the previous bullet point. hen

Release 0.3 - 2003-02-14

Type Changes By
Implemented a protocol system for the simple-jndi root's location. The original file location was retrofitted into this and a classpath location was added. A http location is planned hen
If a protocol is unspecified, then resort to v0.2 format, ie) file:// hen
If root is unspecified, then use the classpath hen
The delimiter in the search values is configurable, so people can more easily mimic LDAP hen
